# Файл 8. JavaScript Core, Node.js и Архитектура интерфейсов

## 1. Движок JavaScript и Event Loop (Браузер)

JavaScript — это однопоточный язык программирования. Он может выполнять только одну операцию за раз (имеет один Call Stack). Асинхронность достигается за счет среды выполнения (браузера или Node.js).

- **Call Stack (Стек вызовов):** Структура данных (LIFO — Last In, First Out), которая записывает, где мы находимся в программе. Если мы вызываем функцию, она помещается в верхнюю часть стека. Когда функция завершается, она удаляется из стека.
- **Web APIs:** Среда браузера предоставляет АРI (например, `setTimeout`, DOM-события, `fetch`), которые выполняются за пределами Call Stack (асинхронно).
- **Очередь микрозадач (Microtask Queue):** Имеет **наивысший приоритет** после очистки Call Stack. Сюда попадают:
    - Коллбэки промисов (`.then()`, `.catch()`, `.finally()`).
    - Функции, добавленные через `queueMicrotask()`.
    - _Специфика:_ Event Loop не перейдет к макрозадачам, пока очередь микрозадач не будет полностью пуста. Если микрозадачи бесконечно порождают новые микрозадачи, интерфейс зависнет.

- **Очередь макрозадач (Macrotask / Task Queue):** Имеет **низший приоритет**. Сюда попадают:
    - `setTimeout`, `setInterval`.
    - Обработчики событий (клики, скролл).
    - Сетевые запросы (когда они завершены и их коллбэки готовы к выполнению).


## 2. Специфика Event Loop в Node.js

В Node.js Event Loop реализован на базе библиотеки **libuv** (написанной на C++) и имеет более сложную структуру фаз, чем в браузере.

- **Фазы Event Loop в Node.js (строгий порядок):**
    1. **Timers:** Выполняет коллбэки из `setTimeout()` и `setInterval()`.
    2. **Pending Callbacks:** Выполняет отложенные коллбэки I/O (ввода-вывода) системного уровня.
    3. **Idle, Prepare:** Внутренние фазы движка.
    4. **Poll:** Получает новые события I/O. Исполняет коллбэки I/O. Если очередь пуста, может блокироваться и ждать новых событий (если нет таймеров).
    5. **Check:** Выполняет коллбэки, зарегистрированные через `setImmediate()`.
    6. **Close Callbacks:** Коллбэки закрытия (например, `socket.on('close')`).

- **Микрозадачи в Node.js:** Выполняются **между каждой фазой** Event Loop.
    - `process.nextTick()`: Имеет абсолютный приоритет над любыми другими микрозадачами (включая Promise). Выполняется сразу после завершения текущей операции, до того, как Event Loop перейдет к следующей фазе.

- **Пул потоков (Thread Pool):** libuv по умолчанию имеет 4 фоновых потока (можно увеличить через `UV_THREADPOOL_SIZE`). Они используются для тяжелых операций: работа с файловой системой (`fs`), криптография (`crypto`), DNS-запросы. Это то, что делает Node.js неблокирующим.

## 3. Ключевые концепции JavaScript

- **Контекст выполнения (Execution Context) и `this`:**
    - В обычных функциях `this` определяется тем, _как_ функция была вызвана (контекст вызова).
    - В стрелочных функциях (arrow functions) `this` определяется _лексически_ — берется из того места, где функция была объявлена. Стрелочные функции не имеют своего `this`.

- **Замыкания (Closures):** Это способность функции запоминать и получать доступ к своей лексической области видимости (переменным снаружи), даже когда эта функция выполняется вне своей лексической области видимости.
- **Hoisting (Всплытие):** Механизм, при котором объявления переменных (`var`) и функций (`function() {}`) "поднимаются" в самый верх их области видимости перед выполнением кода. Переменные `let` и `const` также всплывают, но попадают в "Временную мертвую зону" (TDZ) и недоступны до их фактического объявления в коде.

## 4. Паттерны Frontend-архитектуры и Состояния (Дополнения из ментальной карты)

- **Redux Toolkit (RTK) vs Context API vs Zustand:**
    - _Redux Toolkit:_ Стандарт энтерпрайза. Единое иммутабельное глобальное хранилище. Использует однонаправленный поток: Action -> Dispatcher -> Reducer -> Store. Защищает от случайных мутаций, но требует написания большого количества boilerplate-кода.
    - _Context API:_ Встроенный инструмент React. Отлично подходит для данных, которые меняются редко (тема, локаль, авторизация). **Минус:** При изменении значения в Context перерисовываются _все_ компоненты, которые его потребляют, что делает его плохим выбором для часто меняющихся состояний (например, инпутов формы).
    - _Zustand:_ Золотая середина. Решает проблему ререндеров Context API, позволяет селективно подписываться на изменения конкретных переменных в сторе и не требует сложных оберток (провайдеров).
        
- **Atomic Design:** Методология проектирования UI-компонентов. В отличие от FSD (которая делит приложение по бизнес-логике), Atomic Design делит интерфейс от мельчайших деталей к крупным:
    1. _Атомы:_ Базовые HTML-теги (кнопка, инпут, лейбл).
    2. _Молекулы:_ Группы атомов (инпут + кнопка поиска).
    3. _Организмы:_ Сложные блоки интерфейса (хедер, сетка товаров).
    4. _Шаблоны (Templates):_ Скелет страницы без реальных данных.
    5. _Страницы (Pages):_ Шаблоны, заполненные реальными данными из API или стора.
