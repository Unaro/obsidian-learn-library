# Файл 11. Практика Node.js: Streams, Многопоточность и Очереди задач

## 1. Streams (Потоки данных)

- **Проблема:** Попытка прочитать файл размером 2 ГБ целиком в память (например, через `fs.readFile`) приведет к падению процесса Node.js, так как стандартный лимит памяти движка V8 (Heap Size) составляет около 1.4 ГБ.
- **Решение (Streams):** Обработка данных по частям (чанками / chunks), не дожидаясь загрузки всего объема файла. Это позволяет обрабатывать бесконечно большие объемы данных, используя минимум оперативной памяти.
    
- **Типы потоков:**
    1. _Readable (Для чтения):_ Источник данных (например, чтение файла через `fs.createReadStream` или входящий HTTP-запрос).
    2. _Writable (Для записи):_ Приемник данных (например, запись в файл `fs.createWriteStream` или HTTP-ответ клиенту).
    3. _Duplex (Двунаправленные):_ Поддерживают и чтение, и запись (например, TCP-сокеты).
    4. _Transform (Трансформирующие):_ Разновидность Duplex-потоков, где выходные данные вычисляются из входных (например, сжатие данных `zlib` или парсинг CSV на лету).

- **Пайпинг (Piping):** Метод `.pipe()` связывает Readable поток с Writable. Он автоматически управляет скоростью передачи данных (механизм **Backpressure**), гарантируя, что быстрый источник не переполнит память медленного приемника.

## 2. Многопоточность (Worker Threads)

- **Архитектурный вызов:** Node.js идеально масштабируется для тысяч I/O-операций (запросы к БД, сеть) благодаря асинхронности. Но **CPU-bound задачи** (вычисление сложных метрик, парсинг объемных абстрактных синтаксических деревьев — AST, криптография) полностью блокируют Event Loop. Пока идет расчет, сервер не сможет обработать ни один другой входящий запрос.
- **Модуль `worker_threads`:** Позволяет запускать JavaScript-код в параллельных системных потоках, решая проблему блокировки.
    
- **Специфика:**
    - Каждый воркер имеет свой собственный изолированный экземпляр движка V8, свой Event Loop и свой контекст исполнения.
    - В отличие от создания дочерних процессов (`child_process`), воркеры могут делить общую оперативную память через `SharedArrayBuffer`, что делает обмен бинарными данными между потоками экстремально быстрым.        
    - Главный поток и воркеры общаются через систему передачи сообщений (`postMessage` и прослушивание события `on('message')`).

## 3. Очереди сообщений и Фоновые задачи (Message Queues)

- **Назначение:** Отложенная и гарантированная обработка тяжелых задач. Если в систему загружается массив из 100 тыс. строк для аналитики, держать HTTP-соединение открытым нельзя (запрос отвалится по таймауту браузера или балансировщика).
- **Паттерн Producer-Consumer (Издатель-Потребитель):**
    - _Producer:_ API-сервер принимает данные, быстро сохраняет их во временное хранилище, кладет задачу (например, "Распарсить файл ID 123") в очередь и мгновенно отвечает клиенту (HTTP 202 Accepted).
    - _Consumer (Worker-процесс):_ Отдельный независимый процесс Node.js берет задачу из очереди, выполняет тяжелую логику обработки и затем обновляет статус в основной базе данных или отправляет уведомление по WebSocket.

- **Популярные инструменты:**
    - _Redis + BullMQ / Bull:_ In-memory очереди. Очень быстрые, хранят задачи в оперативной памяти Redis. Из коробки поддерживают повторные попытки (retries), отложенные задачи (delayed jobs) и контроль параллелизма (concurrency).
    - _RabbitMQ / Apache Kafka:_ Полноценные брокеры сообщений. Используются для сложной микросервисной архитектуры, когда одно событие должно быть обработано сразу несколькими разными сервисами по сложным правилам маршрутизации.

## 4. Событийно-ориентированная архитектура (Event Emitter)

- **Класс `EventEmitter`:** Фундаментальный паттерн, на котором построено большинство встроенных модулей Node.js (потоки, HTTP-сервер). Позволяет объектам генерировать именованные события, на которые можно "подписывать" функции-обработчики (Listeners).
- **Отличие от коллбэков и промисов:** Промис может разрешиться только один раз (успешно или с ошибкой). `EventEmitter` реализует паттерн "Наблюдатель" (Observer): событие может срабатывать бесконечное количество раз (например, событие `data` при чтении потока), и на него может реагировать множество независимых слушателей одновременно.
## 5. Backend: Микросервисная архитектура

- **Монолит:** Все приложение (клиент, API, работа с БД, фоновые задачи) работает как единый процесс. Плюсы: легко деплоить и тестировать на старте. Минусы: при падении одной функции падает всё приложение; сложно масштабировать отдельные узлы.
    
- **Микросервисы:** Приложение разбито на десятки маленьких, независимых сервисов (например, Сервис Авторизации, Сервис Аналитики, Сервис Email).
    - _Плюсы:_ Можно писать на разных языках (один сервис на Node.js, другой на Python). Если упадет сервис Email, пользователи все равно смогут логиниться. Можно точечно увеличивать мощности только для нагруженных сервисов.
    - _Минусы:_ Огромная сложность инфраструктуры (Docker, Kubernetes), проблемы с распределенными транзакциями, сложность отладки по сети.
