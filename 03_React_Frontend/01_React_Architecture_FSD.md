# Файл 2. Frontend & Архитектура (React, Next.js, FSD, Zustand)

## 1. Next.js (App Router) & React 19

- **React Server Components (RSC):** Компоненты по умолчанию в Next.js App Router. Рендерятся исключительно на сервере. Не отправляют JavaScript-код в клиентский бандл, что уменьшает его размер. Имеют прямой доступ к серверной инфраструктуре (БД, файловая система). Не поддерживают хуки интерактивности (`useState`, `useEffect`, обработчики событий).
- **Client Components:** Активируются директивой `'use client'` в начале файла. Рендерятся на сервере (для начального HTML) и гидрируются на клиенте. Необходимы для интерактивного UI.
- **Паттерны рендеринга:**
    - _SSR (Server-Side Rendering):_ HTML генерируется динамически при каждом входящем запросе.
    - _SSG (Static Site Generation):_ HTML генерируется один раз на этапе сборки (build-time). Самый быстрый способ отдачи контента.
    - _CSR (Client-Side Rendering):_ Основная отрисовка и получение данных происходят в браузере средствами JavaScript.

- **Server Actions:** Асинхронные функции, выполняемые на сервере, которые можно вызывать напрямую из клиентских компонентов (например, для обработки форм), избегая ручного создания API-роутов.
    

## 2. Архитектура Feature-Sliced Design (FSD)

- **Концепция:** Методология проектирования frontend-архитектуры, направленная на жесткую декомпозицию логики для обеспечения масштабируемости и читаемости.
- **Иерархия слоев (сверху вниз):**
    1. `app`: Инициализация приложения (глобальные стили, провайдеры, базовый layout).
    2. `pages`: Композиция страниц из нижележащих слоев.
    3. `widgets`: Самостоятельные и полноценные блоки интерфейса (например, Header, Сетка дашборда).
    4. `features`: Бизнес-сценарии и действия пользователя (например, Форма авторизации, Создание кастомной метрики).
    5. `entities`: Бизнес-сущности (например, User, Metric, Report). Не содержат бизнес-логики, только отображение и базовое состояние.
    6. `shared`: Переиспользуемый код, не привязанный к специфике бизнеса (UI-kit, API-клиент, утилиты).

- **Главное правило FSD:** Слой может импортировать модули только из слоев, расположенных _строго ниже_ него. Импорты "вверх" и перекрестные импорты на одном уровне (между слайсами) запрещены.
- **Public API (`index.ts`):** Каждый модуль (слайс) обязан экспортировать только то, что необходимо внешнему потребителю, инкапсулируя внутреннюю реализацию.

## 3. Управление состоянием (Zustand)

- **Принципы:** Легковесный стейт-менеджер, работающий на основе хуков. В отличие от Redux, минимизирует boilerplate-код (не требует явного создания actions, reducers, dispatchers).
- **Однонаправленный поток данных (Flux):** Состояние иммутабельно, изменения происходят только через функции-сеттеры, определенные внутри самого стора.
- **Работа вне React:** Zustand позволяет получать доступ к состоянию и обновлять его за пределами компонентов React (например, в обычных утилитарных `.ts` файлах или в слое API), что упрощает интеграцию с FSD.

## 4. AST-анализ и вычисления (Абстрактное синтаксическое дерево)

- **Уязвимость `eval()`:** Использование стандартной функции `eval()` для вычисления строковых математических выражений (метрик/формул) является критической уязвимостью XSS, позволяющей выполнить произвольный код в среде пользователя или сервера.
- **Парсинг через AST:** Для безопасного вычисления формула проходит несколько этапов:
    1. _Лексический анализ (Tokenization):_ Разбиение строки на логические единицы (числа, операторы, переменные).
    2. _Синтаксический анализ (Parsing):_ Построение AST — древовидной структуры, отражающей иерархию и приоритет операций выражения.

- **Безопасная оценка (Evaluation):** Итеративный или рекурсивный обход AST. На этом этапе происходит жесткая валидация узлов дерева (разрешены только математические операторы и заранее заданные переменные из контекста), что гарантирует изоляцию и безопасность сложных расчетов (например, с использованием библиотек вроде Math.js).
