# Файл 10. Углубленный Next.js (App Router) и React 19

## 1. Четыре уровня кэширования в Next.js (App Router)

Next.js кэширует данные по умолчанию на всех возможных этапах, чтобы минимизировать затраты ресурсов сервера и ускорить отдачу.

- **1. Request Memoization (Мемоизация запросов):**
    - _Где работает:_ На сервере во время рендера одного маршрута.
    - _Суть:_ Если в дереве компонентов несколько раз вызывается `fetch` с одинаковым URL и параметрами, Next.js выполнит реальный запрос к БД/API только один раз. Остальные вызовы получат результат из памяти.
    - _Срок жизни:_ Только на время обработки одного HTTP-запроса от пользователя.

- **2. Data Cache (Кэш данных):**
    - _Где работает:_ На сервере между разными запросами и разными пользователями.
    - _Суть:_ Результаты `fetch` сохраняются в постоянный кэш сервера. Если два разных пользователя запросят одну и ту же страницу, данные для второго возьмутся из кэша.

- **3. Full Route Cache (Кэш полного маршрута):**
    - _Где работает:_ На сервере.
    - _Суть:_ На этапе сборки (build) Next.js рендерит серверные компоненты и сохраняет результат в виде HTML и RSC Payload (специального бинарного формата React). Это основа статической генерации (SSG).

- **4. Router Cache (Кэш маршрутизатора):**
    - _Где работает:_ В браузере (на клиенте).
    - _Суть:_ Next.js сохраняет RSC Payload посещенных маршрутов и тех, на которые есть ссылки (через `<Link>`), в памяти браузера. Это делает переходы между страницами мгновенными, без запросов на сервер.


## 2. Инвалидация кэша (Как обновить данные)

- **Time-based (По времени):** Использование опции `next: { revalidate: 3600 }` в `fetch`. Кэш обновится автоматически через заданное количество секунд при следующем запросе.
- **On-demand (По требованию):** Ручное сбрасывание кэша, когда данные точно изменились (например, после добавления новой записи в БД).
    - `revalidatePath('/blog')`: Сбрасывает кэш для конкретного URL.
    - `revalidateTag('posts')`: Сбрасывает кэш для всех `fetch` запросов, которые были помечены этим тегом, независимо от их URL.

## 3. Server Actions (Серверные экшены)

- **Суть:** Асинхронные функции, которые выполняются строго на сервере, но могут вызываться напрямую из клиентских компонентов.
- **Преимущества:**
    - Избавляют от необходимости писать отдельные файлы API-роутов (эндпоинты).
    - Обеспечивают "Progressive Enhancement": если передать Server Action в атрибут `action` HTML-формы, форма будет работать и отправлять данные на сервер даже если у пользователя отключен JavaScript в браузере.

- **Применение:** Идеально подходят для мутации данных (создание, обновление, удаление записей в базе данных).

## 4. Новые хуки React 19

React 19 тесно интегрирован с концепцией Server Actions и решает боли работы с формами и асинхронностью на клиенте.
- **`useActionState`:** Хук для управления состоянием серверного экшена. Принимает сам экшен и начальное состояние. Возвращает текущее состояние (например, ошибки валидации или успешный результат), функцию для вызова экшена и статус `isPending` (в процессе ли загрузка).
- **`useFormStatus`:** Хук, который позволяет дочерним компонентам формы узнать, отправляется ли форма в данный момент. _Важно:_ его нужно использовать в компоненте, который находится _внутри_ `<form>`, чтобы, например, заблокировать (disable) кнопку Submit и показать спиннер.
- **`useOptimistic`:** Позволяет реализовать оптимистичный UI.
    - _Как работает:_ Пользователь нажимает "Лайк" -> UI мгновенно обновляется (лайк ставится) с помощью `useOptimistic` -> параллельно уходит запрос на сервер. Если сервер возвращает ошибку, UI автоматически откатывается к предыдущему состоянию. Это создает ощущение моментального отклика приложения.

- **`use()`:** Революционный хук, который позволяет читать значения из Промисов (Promises) или Контекста (Context) прямо в теле компонента.
    - _Главная фишка:_ В отличие от всех остальных хуков (`useState`, `useEffect`), хук `use()` **можно использовать внутри условий (`if`) и циклов**.

## 5. Работа со списками

- **Виртуализация списков (Windowing):** Техника, при которой в DOM рендерятся _только те элементы, которые сейчас видны на экране_
    - _Популярные библиотеки:_ `react-window`, `@tanstack/react-virtual`.

- **React.memo:** Компонент высшего порядка (HOC) от React. Оборачивает функциональный компонент и кэширует его результат. Рендер произойдет заново, _только_ если изменились его пропсы (shallow compare). Идеально для оптимизации карточек внутри огромных списков.
