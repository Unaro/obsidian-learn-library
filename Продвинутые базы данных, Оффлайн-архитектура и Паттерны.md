## 1. PostgreSQL: Под капотом и Продвинутые фичи

- **Индексы (B-Tree):** Индексы — это структуры данных для ускорения поиска. По умолчанию в PostgreSQL используется B-Tree (сбалансированное дерево). Поиск по нему занимает логарифмическое время $O(\log n)$.
    - _Плюсы:_ Радикально ускоряют выборку (SELECT) и сортировку.
    - _Минусы:_ Замедляют запись (INSERT, UPDATE, DELETE), так как при каждом изменении БД вынуждена перестраивать дерево индексов , а также занимают место на диске.

- **JSONB (Binary JSON):** Специальный тип данных в PostgreSQL для хранения полуструктурированных данных. В отличие от обычного текстового `JSON`, `JSONB` хранится в бинарном формате.
    - _Зачем нужен:_ Идеален для хранения динамических атрибутов (например, кастомных настроек пользователя или сырых логов), когда создание отдельных колонок избыточно. Поддерживает создание GIN-индексов для молниеносного поиска по ключам внутри JSON.

- **CTE (Common Table Expressions) / `WITH`:** Конструкция, позволяющая создавать временные именованные наборы результатов (виртуальные таблицы), которые существуют только во время выполнения одного запроса.    
    - _Применение:_ Значительно улучшает читаемость гигантских SQL-запросов. Поддерживает рекурсию (`WITH RECURSIVE`), что идеально для построения деревьев и иерархий (например, дерева комментариев).

## 2. Архитектура Оффлайн-first: Синхронизация IndexedDB

- **Проблема:** Пользователь загрузил Excel-файл, данные сохранились в локальную IndexedDB, пропал интернет. Как безопасно отправить данные на сервер (Node.js/Python) при восстановлении связи?
- **Решение (Стратегия синхронизации):**
    1. _Флаги состояния:_ Каждая запись в IndexedDB имеет поле статуса (например, `sync_status: 'pending' | 'synced' | 'deleted'`).
    2. _Background Sync API / Service Workers:_ Браузер в фоновом режиме отслеживает появление сети. Как только интернет появляется, Service Worker собирает все записи со статусом `'pending'` и отправляет их пакетом (batch) на сервер.
    3. _Разрешение конфликтов:_ Если данные были изменены и на сервере, и на клиенте (оффлайн), чаще всего применяется стратегия _Last Write Wins_ (побеждает последняя запись по timestamp) или пользователю выдается модальное окно для ручного слияния.
## 3. Паттерны проектирования ООП на практике

ООП ставит в центр разработки объекты и их взаимодействие. Знать принципы инкапсуляции, наследования и полиморфизма — обязательно, но на реальном собеседовании спросят про паттерны (GoF).
- **Dependency Injection (Внедрение зависимостей):** * _Суть:_ Класс не должен сам создавать свои зависимости внутри себя. Зависимости передаются ему извне (например, через конструктор).
    - _Польза:_ Архитектура становится слабосвязанной. Позволяет легко писать Unit-тесты, подменяя реальную базу данных на мок-объект (Mock).

- **Strategy (Стратегия):**
    - _Суть:_ Определение семейства алгоритмов, инкапсуляция каждого из них и обеспечение их взаимозаменяемости.
    - _Где применить в твоем опыте:_ Идеально для парсера AST-деревьев (Urban Analytics). У тебя есть общий класс `Calculator`, в который ты динамически подкидываешь стратегии вычисления: `SumStrategy`, `AverageStrategy` или `CustomMetricStrategy`.

- **Singleton (Одиночка):**
    - _Суть:_ Гарантирует, что у класса есть только один экземпляр, и предоставляет глобальную точку доступа к нему.
    - _Применение:_ Классический пример — класс подключения к базе данных (PostgreSQL) или глобальный Store на фронтенде (Zustand под капотом реализует похожий паттерн).


## 4. Поведенческое интервью: Метод STAR

Технические скиллы важны, но на позицию Fullstack также смотрят на твою адекватность и умение решать проблемы. Тебя 100% попросят: _"Расскажи о самой сложной задаче в Urban Analytics"_. Отвечать нужно строго по фреймворку **STAR**:

- **S (Situation - Ситуация):** "На проекте Urban Analytics стояла задача обрабатывать загружаемые Excel-файлы до 100 000 строк. Изначально мы отправляли их на бэкенд для анализа".
- **T (Task - Задача):** "Это приводило к долгому ожиданию ответа и перегрузке сервера. Мне нужно было перенести первичный анализ и хранение на клиентскую часть, чтобы разгрузить бэкенд и ускорить отрисовку дашбордов".
- **A (Action - Действия):** "Я решил отказаться от хранения данных в Redux, так как оперативная память браузера не справлялась. Я внедрил **IndexedDB** для локального хранения сырых данных вне главного потока. Для безопасного вычисления кастомных формул юзеров я написал парсер на основе **AST (Abstract Syntax Tree)**, чтобы исключить уязвимость через функцию `eval()`, а для управления UI-стейтом использовал легковесный **Zustand**".
- **R (Result - Результат):** "В результате браузер перестал зависать при рендеринге огромных таблиц, безопасность вычислений была обеспечена, а нагрузка на сервер снизилась на X%".
