# Командная работа и Лидерство

> **Назначение:** Работа в команде, code review, менторство, разрешение конфликтов, техническое лидерство. Включает best practices и шаблоны.

---

## 1. Code Review

### 1.1. Чек-лист для ревьюера

**Перед review:**
```
□ Понимаю контекст изменения (ticket, description)
□ Проверяю размер PR (< 400 строк идеально)
□ Выделяю 30-60 минут без прерываний
```

**Технический чек-лист:**
```
□ Код решает заявленную задачу
□ Нет дублирования (DRY)
□ Логирование на месте
□ Обработка ошибок
□ Тесты добавлены/обновлены
□ Документация обновлена
□ Нет security issues
□ Производительность не деградировала
```

**Читаемость:**
```
□ Имена переменных понятны
□ Функции делают одно дело
□ Комментарии объясняют "почему", не "что"
□ Сложность приемлемая
```

---

### 1.2. Шаблоны комментариев

**Позитивная обратная связь:**
```
✅ "Отличная идея с кэшированием! Это ускорило загрузку на 40%."
✅ "Чистое решение, особенно понравилась валидация через Zod."
✅ "Хорошо, что добавил тесты на edge cases."
```

**Конструктивная критика:**
```
❌ "Почему ты использовал здесь useEffect?"
✅ "Что думал об использовании useMemo вместо useEffect? 
    Это могло бы избежать лишнего рендера."

❌ "Это неправильно"
✅ "Есть альтернативный подход: ... Что думаешь?"

❌ "Слишком сложно"
✅ "Можно ли упростить эту функцию? 
    Может, разбить на две меньшие?"
```

**Блокирующие vs Неблокирующие:**
```
🔴 BLOCKING (требует исправления):
• Баги
• Security issues
• Нарушение архитектурных принципов
• Отсутствие тестов на критичный код

🟡 NON-BLOCKING (пожелания):
• Нейминг (если понятен)
• Рефакторинг (если работает)
• Оптимизация (если не критично)
```

### 2.2. Структура 1-на-1 встреч

**Еженедельная встреча с менти:**
```
┌─────────────────────────────────────────────────────────┐
│  1-на-1 Шаблон (30-60 минут)                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Check-in (5 мин)                                    │
│  • Как дела?                                            │
│  • Что было хорошего на прошлой неделе?                 │
│                                                         │
│  2. Обзор прогресса (15 мин)                            │
│  • Какие задачи завершил?                               │
│  • С какими трудностями столкнулся?                     │
│  • Что узнал нового?                                    │
│                                                         │
│  3. Планирование (15 мин)                               │
│  • Какие задачи на следующую неделю?                    │
│  • Где нужна помощь?                                    │
│  • Есть ли блокираторы?                                 │
│                                                         │
│  4. Развитие (10 мин)                                   │
│  • Чему хочешь научиться?                               │
│  • Обратная связь по работе                             │
│  • Карьерные цели                                       │
│                                                         │
│  5. Wrap-up (5 мин)                                     │
│  • Summary договорённостей                              │
│  • Вопросы                                              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 3. Разрешение конфликтов

### 3.1. Модель DESC

```
D — Describe (Опиши ситуацию)
"На последних трёх планированиях ты перебивал меня, 
когда я оценивал задачи"

E — Express (Вырази чувства)
"Это создаёт впечатление, что моя оценка не важна, 
и меня это расстраивает"

S — Specify (Предложи решение)
"Можешь дать мне закончить оценку, а потом задать вопросы?"

C — Consequences (Опиши последствия)
"Тогда мы будем эффективнее планировать и избежим 
недопонимания"
```

## 5. Работа с легаси

### 5.1. Стратегии

**Boy Scout Rule:**
```
"Оставляй код чище, чем нашёл"

Маленькие улучшения каждый раз:
• Переименовать переменную
• Вынести в функцию
• Добавить тест
• Обновить комментарий
```

**Strangler Fig Pattern:**
```
Постепенная замена легаси:

1. Создать новый endpoint/компонент
2. Перенаправить часть трафика
3. Сравнить метрики
4. Переключить полностью
5. Удалить старый код

Повторять пока легаси не исчезнет
```

---

### 5.2. Документирование легаси

```markdown
# Legacy Code Documentation

## Контекст
- Когда написано: 2019
- Почему так: Ограничения по времени, POC для клиента
- Кто автор: @developer (уволился)

## Известные проблемы
1. Нет тестов на расчёт скидок
2. Жёсткая связь с payment provider
3. Таймауты не настроены

## План улучшений
- [ ] Добавить тесты на критичные пути
- [ ] Выделить абстракцию для платежей
- [ ] Настроить мониторинг

## Risk Assessment
- Риск изменений: Высокий
- Бизнес-критичность: Критично
- Частота изменений: Редко
```
