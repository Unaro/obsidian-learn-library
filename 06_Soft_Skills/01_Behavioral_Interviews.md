# Поведенческие интервью и Soft Skills

> **Назначение:** Метод STAR, ответы на сложные вопросы, negotiation, управление временем и командная работа. Включает шаблоны ответов и практические примеры.

---

## 1. Поведенческие интервью (Behavioral Interviews)

### 1.1. Метод STAR — детальное руководство

**Что оценивают интервьюеры:**
```
┌─────────────────────────────────────────────────────────┐
│              Оценка по компетенциям                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  🔹 Problem Solving — как решаете сложные задачи        │
│  🔹 Leadership — влияние и ответственность              │
│  🔹 Teamwork — работа в команде                         │
│  🔹 Communication — ясность изложения                   │
│  🔹 Adaptability — гибкость и обучение                  │
│  🔹 Ownership — чувство ответственности                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Шаблон ответа по STAR:**

```
S — Situation (20% времени)
  Контекст, без лишних деталей
  "На проекте X была проблема Y..."

T — Task (10% времени)
  Ваша конкретная задача
  "Мне нужно было сделать Z..."

A — Action (60% времени) ← ОСНОВНАЯ ЧАСТЬ!
  Что ВЫ сделали (не "мы")
  "Я проанализировал... Я предложил... Я реализовал..."

R — Result (10% времени)
  Измеримый результат с цифрами
  "В результате метрика улучшилась на X%..."
```

### 1.2. Топ-15 вопросов с шаблонами ответов

**Вопрос 1:** "Расскажите о самой сложной технической проблеме"

```
S: "На проекте Urban Analytics пользователи загружали Excel до 100K строк. 
    Система отправляла данные на бэкенд для обработки, что вызывало таймауты 
    и перегрузку сервера."

T: "Мне нужно было перепроектировать архитектуру для обработки на клиенте, 
    сохранив безопасность и производительность."

A: "Я сделал три ключевых шага:
    
    1. IndexedDB для хранения — перенёс данные из Redux в браузерную БД,
       что сняло нагрузку на память (~50MB → ~5MB).
    
    2. AST-парсер для формул — написал безопасный парсер вместо eval(),
       исключив XSS-уязвимость.
    
    3. Виртуализация списков — внедрил react-window для рендеринга только 
       видимых строк (10000 → ~50 DOM элементов)."

R: "Время загрузки дашбордов: 8с → 200мс (40x улучшение).
    Нагрузка на бэкенд: -80%.
    Пользователи получили мгновенный отклик."
```

---

**Вопрос 2:** "Как вы разрешали конфликты в команде?"

```
S: "При разработке нового API возник конфликт: бэкенд-команда хотела 
    GraphQL для гибкости, фронтенд-команда настаивала на REST из-за 
    простоты кэширования."

T: "Нужно было найти решение, которое устроит обе команды и будет 
    технически обоснованным."

A: "Я организовал техническую дискуссию:
    
    1. Составил таблицу сравнения (производительность, сложность, 
       кэширование, learning curve).
    
    2. Предложил гибридный вариант: REST для основных данных + 
       GraphQL для сложных отчётов.
    
    3. Создал proof-of-concept для обоих подходов с метриками."

R: "Команды пришли к консенсусу. REST для 80% эндпоинтов, GraphQL 
    для аналитики. Time-to-market не пострадал, обе команды довольны."
```

---

**Вопрос 3:** "Как вы приоритизируете задачи при сжатых дедлайнах?"

```
S: "За неделю до релиза обнаружили критический баг в оплате, одновременно 
    маркетинг запустил промо-кампанию с ожидаемым трафиком x10, и нужно 
    было закрыть квартальные цели по фичам."

T: "Нужно было определить порядок решения задач с ограниченной командой 
    (5 разработчиков)."

A: "Я использовал матрицу Eisenhower + оценку рисков:
    
    1. Оплата (критично/срочно) — 2 разработчика немедленно.
    
    2. Масштабирование под нагрузку (критично/срочно) — 2 разработчика 
       на кэширование и rate limiting.
    
    3. Квартальные фичи (важно/несрочно) — договорился с PM о переносе 
       части на следующий спринт.
    
    4. Ежедневные стендапы 2x в день для синхронизации."

R: "Баг исправлен за 2 дня. Нагрузку выдержали (99.9% uptime). 
    2 фичи перенесены, но бизнес принял это осознанно."
```

---

**Вопрос 4:** "Расскажите о случае, когда вы ошиблись"

```
S: "При оптимизации запроса к БД я добавил индекс на часто обновляемую 
    колонку, не протестировав на write-нагрузке."

T: "Нужно было исправить деградацию производительности после деплоя."

A: "Как только заметили проблему:
    
    1. Немедленно откатил миграцию через 15 минут после деплоя.
    
    2. Проанализировал EXPLAIN ANALYZE — индекс замедлял UPDATE на 300%.
    
    3. Предложил альтернативу: частичный индекс только для активных записей.
    
    4. Добавил в checklist code review: проверять влияние на write-операции."

R: "Простой составил 30 минут. Новый индекс ускорил SELECT на 80% без 
    влияния на UPDATE. Process improvement внедрён в команду."
```

---

**Вопрос 5:** "Как вы работали с трудным коллегой?"

```
S: "Сеньор-разработчик в команде регулярно блокировал PR других 
    разработчиков агрессивными комментариями без конструктивных предложений."

T: "Нужно было улучшить культуру code review без эскалации менеджменту."

A: "Я предпринял несколько шагов:
    
    1. Личная встреча 1-на-1 — объяснил impact на команду и мораль.
    
    2. Предложил шаблон для review: 'Что хорошо + Что улучшить + Вопросы'.
    
    3. Ввёл SLA на review (24 часа) и ротацию ревьюеров.
    
    4. Публично хвалил за конструктивные review на ретроспективах."

R: "Через 2 месяца тон комментариев изменился. Время review сократилось 
    с 3 дней до 1 дня. Текучесть в команде снизилась."
```

### 1.3. Вопросы для разных уровней

**Junior:**
- "Как вы изучаете новые технологии?"
- "Расскажите о проекте, которым гордитесь"
- "Как справляетесь с задачами, где не знаете решения?"

**Middle:**
- "Как менторили младших разработчиков?"
- "Как предлагали улучшения архитектуры?"
- "Как оцениваете задачи перед планированием?"

**Senior:**
- "Как влияли на техническую стратегию?"
- "Как балансировали между идеальным решением и deadline?"
- "Как разрешали технические разногласия между командами?"

### 2.2. Техники тайм-менеджмента

**Time Blocking:**
```
┌─────────────────────────────────────────────────────────┐
│              Пример дня разработчика                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  09:00 - 10:00  │  Deep Work (сложная задача)          │
│  10:00 - 10:30  │  Code Review                         │
│  10:30 - 11:00  │  ☕ Перерыв                          │
│  11:00 - 12:30  │  Deep Work (продолжение)             │
│  12:30 - 13:30  │  🍕 Обед                             │
│  13:30 - 14:00  │  Daily Standup                       │
│  14:00 - 15:30  │  Встречи / Pair Programming          │
│  15:30 - 16:00  │  ☕ Перерыв                          │
│  16:00 - 17:00  │  Документация / Планирование         │
│  17:00 - 17:30  │  Email / Slack / Админ               │
│                                                         │
│  Правила:                                               │
│  • Deep Work без встреч и уведомлений                  │
│  • Контекстное переключение минимизировать             │
│  • Буферное время на непредвиденное                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Pomodoro для разработки:**
```
25 минут фокуса → 5 минут перерыв → повторить 4 раза → 30 минут перерыв

Для программирования адаптировано:
50 минут фокуса → 10 минут перерыв

Приложения:
• Forest — геймификация фокуса
• Toggl — трекинг времени
• Focus To-Do — Pomodoro + задачи
```

### 2.3. Борьба с выгоранием

**Симптомы:**
```
┌─────────────────────────────────────────────────────────┐
│              Burnout Warning Signs                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  🔴 Физические:                                         │
│  • Хроническая усталость                                │
│  • Проблемы со сном                                     │
│  • Частые болезни                                       │
│                                                         │
│  🟡 Эмоциональные:                                      │
│  • Цинизм и раздражительность                           │
│  • Потеря мотивации                                     │
│  • Чувство бесполезности                                │
│                                                         │
│  🟠 Поведенческие:                                      │
│  • Прокрастинация                                       │
│  • Изоляция от команды                                  │
│  • Переработки без результата                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Профилактика:**
```
1. Границы работы:
   • Чёткое начало и конец рабочего дня
   • No meetings дни (например, среда)
   • Отключение уведомлений после работы

2. Восстановление:
   • Минимум 1 полный выходной без кода
   • Отпуск каждые 3-4 месяца
   • Хобби вне IT

3. Поддержка:
   • Регулярные 1-на-1 с менеджером
   • Peer support в команде
   • Профессиональная помощь при необходимости
```

## 3. Командная работа

### 3.1. Feedback Culture

**Модель SBI для обратной связи:**
```
Situation — Конкретная ситуация
Behavior — Наблюдаемое поведение
Impact — Влияние на команду/проект

Пример позитивного feedback:
"На код-ревью вчера (S) ты предложил оптимизацию запроса (B), 
это ускорило загрузку страницы на 40% (I). Отличная работа!"

Пример корректирующего feedback:
"На стендапе сегодня (S) ты перебил трёх коллег (B), 
это создало ощущение, что их мнение не ценится (I). 
Давай обсудим, как улучшить коммуникацию."
```

**Правила code review:**
```
✅ Делать:
• Начинать с позитива
• Задавать вопросы, не утверждать
• Предлагать альтернативы
• Реагировать быстро (24h SLA)

❌ Не делать:
• Личные комментарии
• "Почему ты сделал..." → "Что думал о..."
• Блокировать из-за предпочтений
• Игнорировать PR больше 2 дней
```

## 4. Negotiation (Переговоры)

### 4.1. Техники для разработчиков

**С дедлайнами:**
```
❌ "Это невозможно сделать к пятнице"
✅ "Мы можем сделать к пятнице, но:
   - Либо сократить функционал до X
   - Либо добавить ещё одного разработчика
   - Либо перенести Y на следующую неделю
   
   Что приоритетнее?"
```

**С техническим долгом:**
```
❌ "Нужно переписать всё с нуля"
✅ "Технический долг замедляет разработку на 30%. 
   Предлагаю выделить 20% времени каждого спринта 
   на рефакторинг. Через 2 месяца скорость 
   восстановится."
```

**С зарплатой:**
```
Подготовка:
• Исследование рынка (levels.fyi, glassdoor)
• Список достижений с цифрами
• Альтернативные предложения (если есть)

Разговор:
"За последний год я:
• Реализовал X, что дало Y% рост метрики
• Менторил 2 junior разработчиков
• Взял на себя ответственность за Z

Исходя из рынка и моего вклада, 
я ожидаю компенсацию в диапазоне [X-Y]"
```
