# Файл 4. Инфраструктура, Сети и Принципы проектирования

## 1. Инфраструктура и Контейнеризация (Docker & Linux)

- **Docker:** Платформа для контейнеризации, упаковывающая приложение и все его зависимости (код, библиотеки, настройки) в стандартизированный контейнер. Позволяет запускать приложение в изолированной среде на любой машине.
- **Образ (Image) vs Контейнер (Container):**
    - _Образ:_ Статический, неизменяемый шаблон или "инструкция" для создания контейнера. Создается на основе `Dockerfile`.
    - _Контейнер:_ Запущенный, "живой" экземпляр образа, работающий в изолированной среде со своим пространством процессов и сетью.

- **Dockerfile:** Текстовый файл с пошаговыми инструкциями (командами) для сборки Docker-образа (`FROM`, `WORKDIR`, `COPY`, `RUN`, `CMD`).
- **Docker Compose:** Инструмент для описания и запуска многоконтейнерных приложений через конфигурационный файл `docker-compose.yml`. Позволяет одной командой поднять связанные сервисы (например, сервер + БД), настроив их порты, переменные окружения и сети.
- **SSH (Secure Shell):** Протокол для безопасного удаленного управления серверами. Обеспечивает шифрование трафика и авторизацию (обычно по криптографическим ключам), используется для базового развертывания сервисов.

## 2. Системы контроля версий (Git)

- **Коммит (Commit):** Уникальный снимок (snapshot) состояния файлов проекта в определенный момент времени. Имеет уникальный SHA-1 хеш, автора, дату и комментарий.
- **`git merge` vs `git rebase`:**
    - `merge`: Создает новый коммит слияния, объединяющий историю двух веток. Сохраняет исходную историю коммитов без изменений.
    - `rebase`: Переписывает историю, перенося коммиты текущей ветки поверх целевой. Создает линейную историю без коммитов слияния.

- **Git Flow:** Методология ветвления. Основные ветки: `master`/`main` (стабильный релизный код), `develop` (интеграция фич), `feature` (новые функции), `release` (подготовка к релизу), `hotfix` (срочные исправления багов в проде).

## 3. Сети, HTTP и Авторизация

- **HTTP (HyperText Transfer Protocol):** Протокол передачи данных по принципу клиент-сервер. Является _Stateless_ (без сохранения состояния) — каждый запрос независим, сервер не хранит контекст между запросами.
- **Методы HTTP и Идемпотентность:**
    - _Идемпотентные_ (повторный запрос не меняет состояние сервера): `GET` (получение данных), `PUT` (полное обновление/замена ресурса), `DELETE` (удаление), `HEAD`.
    - _Не идемпотентные_: `POST` (создание нового ресурса), `PATCH` (частичное обновление).

- **REST (Representational State Transfer):** Архитектурный стиль веб-сервисов. Принципы: взаимодействие через ресурсы (URI), использование стандартных HTTP-методов, обмен данными в форматах JSON/XML, Stateless-взаимодействие.
- **CORS (Cross-Origin Resource Sharing):** Механизм безопасности браузера, блокирующий запросы к ресурсам на другом домене (origin), если сервер явно не разрешил их через CORS-заголовки.
- **JWT (JSON Web Token):** Компактный формат передачи данных об авторизации. Состоит из трех частей, разделенных точками: `Header` (тип и алгоритм), `Payload` (полезная нагрузка: ID пользователя, срок действия), `Signature` (подпись для проверки подлинности).

## 4. Системный анализ и Моделирование
- **BPMN (Business Process Model and Notation):** Нотация для моделирования бизнес-процессов. Ориентирована на последовательность событий во времени, показывает алгоритм действий, исполнителей (через пулы и дорожки), логические шлюзы.
- **IDEF0:** Методология функционального моделирования. Описывает систему как набор взаимосвязанных функций. Не привязана ко времени. Каждый блок имеет 4 стороны: Вход (данные для обработки), Выход (результат), Управление (правила/ограничения), Механизм (кто/что выполняет).
- **UML (Unified Modeling Language):** Язык визуального моделирования для проектирования программных систем (диаграммы классов, вариантов использования, последовательностей).

## 5. Архитектурные паттерны и Принципы разработки

- **DRY (Don't Repeat Yourself):** Избегание дублирования кода. Логика должна быть определена в одном месте.
- **KISS (Keep It Simple, Stupid):** Решения должны быть максимально простыми и минималистичными. Уменьшает вероятность ошибок и упрощает поддержку.
- **YAGNI (You Aren't Gonna Need It):** Не реализуйте функционал, пока он не понадобится явно (отказ от избыточной архитектуры "на будущее").
- **SOLID:**
    - _S (Single Responsibility):_ Класс имеет только одну причину для изменения (одну зону ответственности).
    - _O (Open/Closed):_ Программные сущности открыты для расширения, но закрыты для модификации исходного кода.
    - _L (Liskov Substitution):_ Объекты подклассов могут заменять объекты суперклассов без поломки логики.
    - _I (Interface Segregation):_ Много узкоспециализированных интерфейсов лучше одного громоздкого "универсального".
    - _D (Dependency Inversion):_ Модули зависят от абстракций (интерфейсов), а не от конкретных реализаций.

- **MVC (Model-View-Controller):** Архитектурный паттерн. 
	- `Model` — бизнес-логика и работа с БД; 
	- `View` — пользовательский интерфейс; 
	- `Controller` — обработка ввода и связующее звено между Model и View.
