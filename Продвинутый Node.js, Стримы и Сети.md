# Файл 17. Продвинутый Node.js, Стримы и Сети

## 1. Стримы (Streams) и Буферы

- **Проблема:** Чтение тяжелого файла (например, CSV на 2 ГБ) через `fs.readFile()` загружает весь файл в оперативную память, что приведет к падению сервера с ошибкой _Out of Memory_.
    
- **Streams (Потоки):** Позволяют читать и записывать данные порциями (чанками/chunks). Память не переполняется.
    
- **Типы потоков:**
    1. _Readable:_ Источник данных (чтение файла, входящий HTTP-запрос).
    2. _Writable:_ Приемник данных (запись в файл, HTTP-ответ).
    3. _Duplex:_ И чтение, и запись (TCP-сокеты).
    4. _Transform:_ Duplex-поток, который модифицирует данные на лету (например, архивация `zlib` или парсинг CSV).

- **Backpressure (Обратное давление):** Механизм при связывании потоков через `.pipe()`. Если Readable поток читает данные быстрее, чем Writable успевает их записывать, `pipe` автоматически приостановит чтение, пока приемник не освободится.

## 2. Event Loop в Node.js (Отличие от браузера)

**Event Loop (Цикл событий)** — это бесконечный цикл, который координирует работу Call Stack и очередей задач. Его главная и единственная задача: **смотреть на Call Stack и на очереди задач**. Если Call Stack пуст, Event Loop берет первую задачу из очереди и помещает её в Call Stack для выполнения.

Чтобы понять Event Loop, нужно знать, из чего состоит среда выполнения:
- **Call Stack (Стек вызовов):** Структура данных (LIFO — последним пришел, первым ушел), в которой хранятся все выполняемые в данный момент синхронные функции. Если функция вызывает другую функцию, та добавляется поверх стека.
- **Куча (Heap):** Область памяти, где хранятся переменные и объекты.
- **libuv (C++ библиотека):** Предоставляет пулы потоков (Thread Pool) для выполнения тяжелых операций (например, работа с файловой системой, сетью, криптографией) вне главного потока JS. Именно она делает Node.js асинхронным.
- **Очереди задач (Task Queues):** Места, куда попадают коллбэки (функции обратного вызова) после того, как асинхронная операция (например, таймер или запрос к БД) завершилась.

В Node.js есть две основные очереди, и Event Loop отдает им разный приоритет:
- **Microtask Queue (Очередь микрозадач):** Имеет **наивысший приоритет**. Сюда попадают коллбэки от `Promises` (методы `.then()`, `.catch()`, `.finally()`) и `process.nextTick()`. Event Loop всегда очищает очередь микрозадач полностью, прежде чем перейти к макрозадачам.
- **Macrotask Queue (Очередь макрозадач):** Сюда попадают коллбэки от `setTimeout`, `setInterval`, `setImmediate`, и операции ввода-вывода (I/O).

В Node.js цикл событий разбит на строгие фазы (на базе библиотеки `libuv`)
1. **Timers (Таймеры):** Выполняет коллбэки, запланированные через `setTimeout()` и `setInterval()`.
2. **Pending Callbacks (Ожидающие коллбэки):** Выполняет коллбэки операций ввода-вывода, отложенные на следующую итерацию цикла (например, ошибки TCP-соединений).
3. **Idle, Prepare:** Используются только внутри самого Node.js (для внутренних нужд).
4. **Poll (Опрос):** Самая важная фаза. Здесь Node.js получает новые события ввода/вывода (I/O) и выполняет их коллбэки (например, чтение файла завершено, пришел HTTP-запрос). Если очередь Poll пуста, Node.js может "заснуть" и ждать новых событий.
5. **Check (Проверка):** Здесь выполняются коллбэки функции `setImmediate()`.
6. **Close Callbacks (События закрытия):** Выполняются коллбэки закрытия соединений (например, `socket.on('close', ...)`).

- **Микрозадачи:** Выполняются _между_ фазами Event Loop. При этом `process.nextTick()` имеет абсолютный приоритет и выполняется раньше промисов.

## 3. Worker Threads (Многопоточность в Node.js)

Node.js написан на C++ и JavaScript. Главная особенность Node.js заключается в том, что он **однопоточный** (выполняет один блок кода JavaScript за раз), но при этом **асинхронный и неблокирующий**. Это достигается как раз за счет механизма Event Loop (цикла событий) и библиотеки **libuv**.

- **Концепция:** Для тяжелых математических вычислений (CPU-bound), таких как парсинг AST-дерева, Node.js использует модуль `worker_threads`.
- **Принцип:** Создается параллельный поток со своим изолированным Event Loop. Главный поток не блокируется и продолжает обслуживать других пользователей. Воркеры могут делить память через `SharedArrayBuffer` для сверхбыстрой передачи данных.

## 4. Real-Time протоколы связи

- **WebSockets:** Двунаправленное постоянное соединение (Full-Duplex). Клиент и сервер могут отправлять сообщения друг другу в любой момент. (Идеально для чатов, онлайн-игр).
- **SSE (Server-Sent Events):** Однонаправленное соединение (только Сервер -> Клиент). Работает поверх обычного HTTP. (Идеально для пуш-уведомлений, стриминга котировок или статуса загрузки файла).
- **Long Polling:** Клиент отправляет запрос, а сервер не отвечает до тех пор, пока не появятся новые данные. После ответа клиент тут же отправляет новый запрос. (Используется как fallback-вариант, если сокеты заблокированы прокси-сервером).
