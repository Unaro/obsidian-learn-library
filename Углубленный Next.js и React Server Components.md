# Файл 16. Углубленный Next.js и React Server Components

## 1. React Server Components (RSC) vs SSR

- **SSR (Server-Side Rendering):** Сервер генерирует HTML и отправляет его клиенту вместе с JavaScript-бандлом компонентов. Затем на клиенте происходит _гидратация_, чтобы сделать HTML интерактивным.
    
- **RSC (Серверные компоненты):** Исполняются _исключительно_ на сервере. Их код и зависимости (библиотеки) **никогда не попадают в клиентский JS-бандл**, что радикально уменьшает его вес. Они передаются на клиент в виде специального сериализованного формата (RSC Payload), который React использует для обновления DOM.
    

## 2. Гидратация (Hydration) и её проблемы

- **Концепция:** Процесс "оживления" статического HTML, пришедшего с сервера. React пробегается по DOM-дереву и навешивает на него обработчики событий (`onClick` и т.д.).
    
- **Hydration Mismatch (Ошибка несовпадения):** Возникает, когда HTML, сгенерированный на сервере, отличается от того, что React пытается отрендерить при первой загрузке на клиенте.
    
    - _Пример ошибки:_ Использование `Math.random()` или `Date.now()` прямо в разметке компонента (сервер сгенерирует одно время, а браузер при гидратации — другое).
        
    - _Решение:_ Использовать хук `useEffect` (он срабатывает только на клиенте после гидратации) для рендера динамических клиентских данных.
        

## 3. Четыре уровня кэширования в Next.js (App Router)

1. **Request Memoization:** Работает на сервере в рамках _одного запроса_. Если в разных компонентах 5 раз вызывается `fetch` с одинаковым URL, Next.js сделает реальный запрос только 1 раз.
    
2. **Data Cache:** Персистентный кэш `fetch`-запросов на сервере. Сохраняет ответы от внешних API между разными пользователями и деплоями. Инвалидируется через `revalidatePath` или `revalidateTag`.
    
3. **Full Route Cache:** Относится к SSG (Static Site Generation). HTML и RSC Payload статических страниц кэшируются сервером на этапе сборки (build).
    
4. **Router Cache:** Клиентский кэш в браузере. Next.js запоминает RSC Payload посещенных страниц, чтобы при навигации (через `<Link>`) переходы были мгновенными и не требовали запроса к серверу.
    

## 4. Server Actions

- **Назначение:** Асинхронные функции, которые выполняются на сервере, но могут быть вызваны напрямую из клиентских компонентов (например, при сабмите формы).
    
- **Плюсы:** Избавляют от необходимости вручную писать эндпоинты (API Routes), писать логику `fetch`-запросов на клиенте и обеспечивают строгую типизацию между клиентом и сервером.
