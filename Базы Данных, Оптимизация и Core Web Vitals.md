# Файл 18. Базы Данных, Оптимизация и Core Web Vitals

## 1. Базы данных: Проблема N+1

- **Суть проблемы:** Возникает при использовании ORM. Когда мы делаем 1 запрос на получение списка пользователей (N пользователей), а затем в цикле для каждого пользователя делаем еще 1 запрос, чтобы получить его посты. В итоге вместо 1 запроса мы делаем N+1 запросов, что убивает базу данных.
- **Решение на уровне БД:** Использование SQL-оператора `JOIN`, чтобы получить все связанные данные за один проход. В Drizzle ORM это решается через `with: { posts: true }`.
- **Решение на уровне кода (Node.js):** Паттерн **DataLoader**. Он собирает все одиночные ID в массив и делает один пакетный (batch) запрос к БД (`WHERE id IN (...)`).

## 2. Индексы (B-Tree) под капотом

- **B-Tree (Сбалансированное дерево):** Структура, по которой БД строит индексы. Поиск по ней занимает логарифмическое время $O(\log n)$, что позволяет найти запись среди миллионов строк за миллисекунды.
- **Компромисс (Trade-off):** Почему нельзя проиндексировать вообще все колонки? Индексы ускоряют чтение (`SELECT`), но замедляют запись (`INSERT`, `UPDATE`, `DELETE`). При каждом изменении данных базе приходится заново перестраивать и балансировать дерево индексов.

## 3. Транзакции и ACID на практике

Если процесс прерывается посередине банковского перевода (деньги списались у одного, но не дошли другому) — это катастрофа.

- **Транзакция:** Логическая единица работы. Гарантирует выполнение по правилу "Всё или ничего".

- **Команды:**
    - `BEGIN` — начало транзакции.
    - `COMMIT` — успешное применение всех изменений к базе.
    - `ROLLBACK` — откат всех изменений, если хотя бы один шаг завершился с ошибкой.

## 4. Frontend Performance: Core Web Vitals

Метрики Google, по которым оценивается UX приложения (важно для SEO и конверсии):

- **LCP (Largest Contentful Paint):** Время отрисовки самого крупного элемента в видимой области экрана (герой-картинка, заголовок). Хороший показатель — до 2.5 секунд.
- **INP (Interaction to Next Paint):** Время реакции интерфейса. Замеряет задержку между кликом пользователя и визуальным откликом на экране (пришел на замену FID).
- **CLS (Cumulative Layout Shift):** Накопительный сдвиг макета. Оценивает визуальную стабильность. Возникает, когда картинка загрузилась позже и "сдвинула" текст, который пользователь уже начал читать. Бороться с этим помогают заданные размеры `width`/`height` для медиа-контента и паттерн Shimmer UI.
