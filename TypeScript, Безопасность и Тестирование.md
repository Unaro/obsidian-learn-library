# Файл 7. Дополнительный стек: TypeScript, Безопасность и Тестирование

## 1. TypeScript Core

- **Статическая типизация:** TypeScript добавляет строгую типизацию в JavaScript на этапе компиляции. Это позволяет отлавливать ошибки (например, обращение к несуществующему свойству объекта) до запуска кода в браузере или на сервере.
- **`interface` vs `type`:**
    - _`interface`_: Используется для описания формы объектов. Легко расширяется (extends) и поддерживает "слияние" (declaration merging) — если объявить два интерфейса с одним именем, они объединятся.
    - _`type` (Type Alias)_: Более универсален. Может описывать примитивы, объединения (Union: `type Status = 'ok' | 'error'`), пересечения (Intersection: `&`) и кортежи. Не поддерживает слияние.

- **Generics (Дженерики):** Переменные для типов (`<T>`). Позволяют создавать переиспользуемые компоненты и функции, которые работают с различными типами данных, сохраняя при этом строгую типизацию (например, типизация ответа от API: `Response<UserData>`).
- **Utility Types (Утилитарные типы):**
    - `Partial<T>`: Делает все свойства типа `T` необязательными (опциональными).
    - `Omit<T, 'K'>`: Создает новый тип, исключая свойство `K` из типа `T`.
    - `Pick<T, 'K'>`: Создает новый тип, оставляя только свойство `K` из типа `T`.
    - `Record<K, T>`: Создает тип словаря (объекта), где ключи имеют тип `K`, а значения — тип `T`.

## 2. Web-Безопасность (OWASP)

- **XSS (Cross-Site Scripting):** Уязвимость, при которой злоумышленник внедряет вредоносный JavaScript-код в страницу, просматриваемую другими пользователями.
    - _Защита:_ Экранирование данных при выводе (React делает это по умолчанию, кроме `dangerouslySetInnerHTML`), использование Content Security Policy (CSP), безопасный парсинг данных (например, через AST вместо `eval()`).

- **SQL Injection (SQLi):** Внедрение вредоносного SQL-кода через поля ввода для манипуляций с базой данных (чтение, удаление).
    - _Защита:_ Использование параметризованных запросов или ORM (например, Drizzle, Django ORM), которые автоматически экранируют входящие переменные и отделяют логику запроса от данных.

- **CORS (Cross-Origin Resource Sharing):** * _Механизм:_ Браузер блокирует AJAX-запросы к доменам, отличным от того, с которого загружена страница.
    - _Preflight Request:_ Для сложных запросов (например, POST с кастомными заголовками) браузер сначала отправляет предварительный запрос с методом `OPTIONS`, чтобы узнать, разрешает ли сервер основную операцию.
    - _Заголовки:_ Сервер должен ответить заголовком `Access-Control-Allow-Origin: *` (или указать конкретный домен).


## 3. Пирамида тестирования

- **Unit (Модульное тестирование):** Тестирование изолированных участков кода (отдельных функций, методов класса, простых React-компонентов без зависимостей). Самые быстрые и массовые тесты. Инструменты: Jest, Vitest.
- **Integration (Интеграционное тестирование):** Проверка взаимодействия между несколькими модулями (например, компонент + Zustand store, или контроллер API + база данных).
- **E2E (End-to-End / Сквозное тестирование):** Тестирование всего приложения целиком с точки зрения конечного пользователя в реальном браузере. Проверяется весь флоу (от клика по кнопке до изменения в БД). Самые медленные и дорогие в поддержке тесты. Инструменты: Cypress, Playwright.

## 4. Стилизация: TailwindCSS

- **Utility-first парадигма:** Вместо написания кастомного CSS в отдельных файлах, стилизация происходит путем добавления множества мелких, специализированных классов (utilities) прямо в HTML/JSX разметку (например, `flex items-center p-4 bg-red-500`).
- **Плюсы:** * Нет необходимости придумывать имена для классов (решает проблему методологии BEM).
    - Отсутствие переключения контекста между файлами `.tsx` и `.css`.
    - Глобальные стили не ломаются при изменениях (каждый класс изолирован).

- **Оптимизация (JIT-компилятор):** Tailwind не загружает весь свой огромный словарь классов в итоговый бандл. JIT (Just-In-Time) компилятор сканирует исходный код во время сборки и генерирует CSS _только_ для тех классов, которые реально были использованы, что делает итоговый файл стилей экстремально малым (часто менее 10kb).

## 5. Type Guarding и Narrowing

- **Type Narrowing (Сужение типов):** Процесс, при котором TypeScript понимает, что тип переменной стал более конкретным внутри блока кода (например, внутри `if`).
    
- **Type Guards (Защитники типов):** Способы сузить тип:
    - _Оператор `typeof`:_ `if (typeof value === 'string') { ... }`
    - _Оператор `in`:_ Проверка наличия свойства в объекте `if ('error' in response) { ... }`
    - _Кастомные Type Guards (Type Predicates):_ Функция, возвращающая `value is Type`.

```TypeScript
	function isUser(obj: any): obj is User {
	  return obj !== null && typeof obj === 'object' && 'username' in obj;
	}
```
